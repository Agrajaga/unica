# Platform 8.3.27 / export format 2.20 implementation plan

## Status

This plan records the approved fixed-profile implementation and its current
execution state. It replaces the earlier assumption that this iteration would
expose native migration operations.

The current scope is:

- one writable profile: platform `8.3.27`, export format `2.20`;
- centralized owner/version preflight for modifying platform-XML workflows;
- read-only warnings for unsupported source formats;
- explicit user-driven load and re-export with 1C 8.3.27 tooling for older
  sources;
- no automatic or native migration;
- no downgrade of newer formats;
- document-family corrections only where evidence supports them.

Native migration and multiple format profiles are deferred future work. They
are not incomplete tasks in this implementation plan.

This plan is the prerequisite fixed-writer/profile phase of issue #126. Passing
its corpus does not complete the broader optional capability-driven XDTO/XSD
service proposed by that issue; that service is a separate design phase.

## Source of truth and constraints

- Official 1C 8.3.27 documentation maps the platform line to export format
  `2.20`.
- Missing version on a version-owning root means `1.0`.
- A source below `2.20` is read-only in Unica until the user explicitly loads
  and re-exports it with 1C:Enterprise 8.3.27 tooling.
- A source above `2.20` is read-only and unsupported for mutation; no downgrade
  is offered. The warning states that 1C 8.5 is not supported yet but is planned.
- Subordinate XML is not assumed to own a version. Resolve the real same-case
  source-set or standalone owner first.
- Raw XSD is additional evidence; real 8.3.27 platform load/roundtrip is final
  compatibility proof.
- Do not commit `docs-local`, schema corpora, credentials, disposable
  infobases, generated corpus directories, or platform staging trees.
- Keep the public MCP boundary as one server named `unica` with typed
  `unica.*` tools.

## Delivered architecture

### Active profile and classifier

- [x] `domain/format_profile.rs` owns platform `8.3.27` and export format
  `2.20`.
- [x] Numeric component comparison classifies missing/older, supported, newer,
  and malformed versions without floating-point comparison; only the exact raw
  lexical slice `2.20`, before entity decoding, is supported. Numeric
  equivalents such as `2.20.0` and entity spellings such as `2.&#50;0` are
  invalid.
- [x] Native writers and validators no longer control the contract with local
  version allowlists or legacy fallbacks.

### Central application preflight

- [x] `format_guard.rs` runs before support checks and native handlers.
- [x] Operation descriptors declare `ExistingDump`, `OptionalExistingBase`,
  `NewDump`, or `None` plus effective path resolution policy.
- [x] Incompatible mutations return structured diagnostics and no writes.
- [x] Read-only operations receive the same compatibility classification as a
  warning and continue where parsing is safe.
- [x] New CF/EPF/ERF output uses the active profile; CFE initialization guards
  an optional base before output creation.

### Real platform-XML owner

- [x] CF/CFE resolve source-set `Configuration.xml`.
- [x] EPF/ERF resolve their sibling top-level artifact descriptor.
- [x] Recognized standalone version-owning roots are classified directly.
- [x] Versionless MXL/DCS/ClientApplicationInterface content inherits a
  same-case source-set owner where applicable rather than being treated as
  format `1.0` by itself.
- [x] Path identity is normalized and bounded by the configured source set or
  workspace.
- [x] Missing, unreadable, malformed, wrong-QName, and ambiguous owners produce
  `formatVersionInvalid` before mutation.
- [x] Handler aliases, default `SrcDir/ObjectName` targets, and multi-input form
  compilation paths are covered by an exhaustive descriptor regression.
- [x] Existing Form, DCS, and MXL targets are checked by exact QName before
  mutation even for `.XML` and suffixless paths; correct versionless DCS/MXL
  roots and genuinely absent outputs remain valid.

### Older/newer user contract

- [x] Older owners block mutation and leave source bytes unchanged.
- [x] Older read-only calls continue with a compatibility warning.
- [x] Every owner kind recommends explicit load and re-export using platform
  1C 8.3.27 and states that Unica does not migrate automatically.
- [x] No diagnostic names a nonexistent public operation.
- [x] Newer owners block mutation, retain read-only warnings, state that 1C 8.5
  is not supported yet but is planned, and never recommend downgrade.

The existing machine-readable code `formatMigrationAvailable` means that a
manual remediation path exists. It is not a callable endpoint.

## Current TDD and verification gates

### Format boundary

- profile classification: missing/older, exact literal `2.20`, newer,
  malformed, numerically equal non-exact spellings, and entity-spelled values;
- every mutating platform-XML descriptor has an effective path policy;
- older/newer/malformed owners block before handler/write;
- read-only aliases warn and continue;
- owner warnings for Configuration, Extension, EPF, ERF, and standalone XML
  contain the manual 1C 8.3.27 re-export guidance and no invented tool name.

### Writers and document families

- all version-owning writer regressions expect `2.20`;
- MXL and ExchangePlan fixture tests compare exact normalized platform output;
- wrong MXL/Form/DCS roots fail before optional output or mutation;
- Form QName bindings are validated and emitted bindings are rechecked;
- `cf.init` public-call and two-cycle platform tests prove the minimal scaffold;
- corpus manifest/hash validation binds regular files, non-XML payloads, public
  calls, owner links, and exact empty-directory topology;
- `support.edit` binds one parsed binary preimage, all `.cf` preimages, and
  case-insensitive directory membership to one transaction.

## Deferred future work

### Native migration

A native migration workflow may be designed later only after explicit approval.
It would require a new public API contract, a verified platform selector,
isolated staging, load/re-export/reload validation, an atomic publication and
recovery protocol, cancellation behavior, security review, and provenance that
does not leak secrets. None of those operations, files, skills, or tests exists
in the current supported contract.

Until then, the supported remediation for an older source is manual and
user-authorized: load it with 1C:Enterprise 8.3.27 tooling, re-export it as
`2.20`, validate it, and retry the Unica mutation.

### Multiple format profiles

Supporting 1C 8.5 or another platform line requires a separately approved
profile resolver and per-family evidence. A future profile must not weaken the
current `8.3.27 / 2.20` no-write boundary and must not infer a platform version
solely from an XML format number.
